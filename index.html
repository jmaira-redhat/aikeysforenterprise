Your engineers are already using Claude Code. Some are using personal API keys. Some have accidentally committed a .env file and quietly rotated it before anyone noticed. Most spend 20–40 minutes on day one configuring credentials instead of writing code.

This isn't a future risk. It's happening right now, across your fleet, at every onboarding. Project Shield fixes all three problems with one architectural change.

The bottleneck

One root cause, three flavors of pain

The default Claude Code setup hands every developer a static API key and says "manage this yourself." That singular decision introduces friction for the developer and risk for the organization.

Day one

Onboarding friction

New hires and contractors spend their first productive hours on credential setup — not shipping. Project leads become impromptu IT support desks.

Ongoing

Flow state tax

Developers carry background anxiety about accidentally committing a .env file. That cognitive overhead is a real, measurable drag on output quality.

Offboarding

Credential sprawl

When a contractor wraps up, their key is still on a personal laptop. Your CISO's audit nightmare is already a live spreadsheet problem.

CISO View

Zero Trust violation — by default

Static, plain-text API keys on developer endpoints directly contradict Assume Breach principles. Project Shield moves all credential authority into the OpenShift cluster. No key ever touches a developer's disk — the endpoint is eliminated as an attack surface entirely.

The architecture

Project Shield in three phases

Instead of managing credentials per-developer, Project Shield centralizes them once — then the DevWorkspace Controller automatically injects them into every Dev Spaces container at launch. Developers open their workspace and Claude Code is ready. No keys. No config. No anxiety.

1
Enterprise Vault CyberArk · HashiCorp Vault
The raw API key lives only here — outside the cluster entirely. Admins never copy or paste it. Full audit log, access policies, and secret lifecycle management at the source.
Source of truth
2
ClusterSecretStore ESO operator resource
Establishes a continuously authenticated, verified bridge between OpenShift and the vault. Trust is never assumed — every sync cycle re-authenticates. The cluster never holds a long-lived vault token.
Verified bridge
3
External Secrets Operator ExternalSecret resource
Watches the vault and auto-syncs changes into a native OpenShift Secret. When the vault rotates a key, ESO propagates it to all active workspaces within minutes — zero manual steps, zero downtime.
Auto-rotation engine
4
OpenShift Secret ESO-owned, labeled
The only Secret in the cluster — and it is owned and managed by ESO, not by a human. Labeled mount-to-devworkspace=true so the DevWorkspace Controller can mount it. Ephemeral: if ESO stops syncing, it expires.
Ephemeral & owned
5
Dev Spaces Pod Runtime injection only
The DevWorkspace Controller mounts the secret into the pod's runtime environment only — never to the filesystem. The developer sees ✓ Shielded. No key is persisted anywhere a developer can read or copy it.
Runtime only
🛡

A plain OpenShift Secret stops at layer 4 — useful, but static and manually managed. The ESO + ClusterSecretStore chain means the cluster never holds a credential that wasn't freshly authorized by the vault. That's the Zero Trust difference.

01
Admin
Once

Admin configures the vault + ESO bridge — once

A cluster admin sets up the ClusterSecretStore — a continuously authenticated connection from OpenShift to your enterprise vault (CyberArk, HashiCorp Vault). The ESO ExternalSecret resource is declared once, pointing at the vault path. From this point forward, the vault is the sole source of truth. The resulting OpenShift Secret is ESO-owned, labeled mount-to-devworkspace=true, and automatically refreshed on every rotation — no human ever touches it again.

02
Dev
One click

Developer onboards with a single command

A project lead shares a devfile.yaml Git URL. The new developer runs oc apply -f ai-app-devfile.yaml — or just clicks a link. Their fully configured, AI-ready Dev Spaces workspace spins up immediately. They have no visibility into the credential chain above them. Zero credential steps. Day-one productivity starts on day one.

03
Shield
Auto

The controller injects the key into the pod — invisibly

The DevWorkspace Controller detects the ESO-managed labeled secret and mounts it into the running Dev Spaces pod as ANTHROPIC_API_KEY — runtime only, never to disk. The developer's terminal shows ✓ Shielded. When the vault rotates the key, ESO syncs it, and the next pod launch picks up the new value automatically. If an endpoint is compromised, there is nothing on it to steal.
